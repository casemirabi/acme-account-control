Visão geral

Guia de instruções da API CLT para integração externa.

Objetivo da API

  • Permitir consultar vínculos empregatícios utilizando o CPF.
  • Iniciar a requisição de consulta de forma segura e rastreável.
  • Acompanhar o processamento por meio de um request_id.
  • Obter o resultado final da consulta.

Fluxo básico

  • O cliente envia o CPF para o endpoint de criação da consulta.
  • A API valida autenticação, formato do CPF e disponibilidade de crédito.
  • A requisição é registrada com um identificador próprio.
  • O cliente acompanha o resultado pelo endpoint de status.

Autenticação

O acesso externo é protegido por chave de API enviada no cabeçalho HTTP.

Header obrigatório

x-acme-key: SUA_CHAVE_DE_API Content-Type: application/json

Regras importantes

  • A chave de API é obrigatória para iniciar a consulta.
  • A chave de API também é obrigatória para consultar o status.
  • Requisições sem autenticação válida retornam erro de acesso.

Fluxo da consulta

Entenda a sequência de ponta a ponta para integração segura.

Etapas

  1. Enviar CPF para o endpoint POST /api-clt.
  2. Receber request_id e status inicial.
  3. Consultar o endpoint GET /api-clt-status.
  4. Acompanhar a consulta até a finalização.

Identificador da consulta

  • request_id: identificador da requisição. É o principal ID para consulta de status.
Recomendação operacional: grave o request_id no seu sistema imediatamente após o POST. Sem ele, o acompanhamento da consulta não poderá ser feito corretamente.

Endpoints da API

Estes são os endpoints públicos utilizados pelo integrador.

Endpoint de requisição
POST /api-clt
Endpoint de status
GET /api-clt-status

Estrutura da requisição

Conteúdo esperado no envio de uma nova consulta CLT.

Body mínimo obrigatório

{ "cpf": "12345678900" }
Campo Tipo Obrigatório Descrição
cpf string Sim CPF com 11 dígitos numéricos, sem máscara.

Regras de envio do CPF

  • Envie somente números.
  • Não envie pontos, traços ou espaços.
  • Se o CPF tiver quantidade inválida de dígitos, a API retorna erro 400.

Exemplo correto

12345678900

Exemplo incorreto

123.456.789-00

Exemplo em cURL

curl --request POST "https://maiscorban.net/api-clt" \ --header "Content-Type: application/json" \ --header "x-acme-key: SUA_CHAVE_DE_API" \ --data '{ "cpf": "12345678900" }'

Exemplo em PowerShell

$headers = @{ "Content-Type" = "application/json" "x-acme-key" = "SUA_CHAVE_DE_API" } $body = @{ cpf = "12345678900" } | ConvertTo-Json Invoke-RestMethod ` -Uri "https://maiscorban.net/api-clt" ` -Method Post ` -Headers $headers ` -Body $body

Consulta de status

Endpoint usado para acompanhar o andamento da consulta previamente criada.

Parâmetro obrigatório

request_id=clt_0ab12c345678

Exemplo de chamada

GET /api-clt-status?request_id=clt_0ab12c345678

Exemplo em cURL

curl --request GET "https://maiscorban.net/api-clt-status?request_id=clt_0ab12c345678" \ --header "x-acme-key: SUA_CHAVE_DE_API"

Exemplo em PowerShell

$headers = @{ "x-acme-key" = "SUA_CHAVE_DE_API" } Invoke-RestMethod ` -Uri "https://maiscorban.net/api-clt-status?request_id=clt_0ab12c345678" ` -Method Get ` -Headers $headers

Status possíveis

Estes são os status relevantes para o integrador no consumo da API.

Status Significado Ação recomendada
pending A requisição foi criada e está aguardando processamento ou retorno. Consultar novamente após alguns segundos.
completed A consulta foi concluída e o resultado já pode ser consumido em response_data. Ler, persistir e encerrar o fluxo.
failed A consulta foi encerrada com falha e contém informações em error. Registrar o erro e decidir se haverá nova tentativa.

Erros e falhas

Estes são os erros mais relevantes para integração e suporte.

HTTP Código Quando acontece
400 INVALID_CPF CPF ausente ou com quantidade inválida de dígitos.
400 MISSING_REQUEST_ID Consulta de status sem o parâmetro request_id.
401 NOT_AUTHENTICATED Chave de API ausente ou inválida.
402 NO_CREDITS Sem créditos disponíveis para realizar a consulta.
404 NOT_FOUND request_id não encontrado.

Exemplo de situação quando não houver créditos

{ "success": false, "error": { "code": "NO_CREDITS", "message": "Sem créditos" } }

Exemplo de erro de autenticação

{ "success": false, "error": { "code": "NOT_AUTHENTICATED", "message": "Você precisa estar logado ou informar uma API key válida." } }

Exemplo de erro de CPF inválido

{ "success": false, "error": { "code": "INVALID_CPF", "message": "CPF inválido. Deve conter 11 dígitos." } }

Exemplo de erro por request_id ausente

{ "success": false, "error": { "code": "MISSING_REQUEST_ID", "message": "request_id obrigatório" } }

Boas práticas de integração

Regras recomendadas para reduzir falhas operacionais e evitar consumo desnecessário de recursos.

  • Armazene o request_id imediatamente após criar a consulta.
  • Faça o acompanhamento com intervalo moderado entre as tentativas.
  • Não dispare várias consultas simultâneas para o mesmo CPF sem necessidade operacional clara.
  • Trate separadamente erro técnico e resultado de negócio.
  • Registre logs da integração no seu sistema, mas nunca grave a chave da API em texto aberto.
Evite interpretar o status apenas pelo texto da mensagem. O comportamento correto é sempre usar os campos estruturados como success, status, error e response_data.

Segurança e observações finais

Regras importantes para uso correto do endpoint público.

  • A chave enviada em x-acme-key deve ser mantida em ambiente seguro.
  • Não exponha a chave em front-end público, aplicativos sem proteção ou logs de navegador.
  • Os resultados podem retornar CPF mascarado no fluxo de status.
Em caso de dúvida de integração, utilize primeiro o fluxo mínimo: criar consulta com CPF válido, guardar o request_id e consultar o status até a finalização.
